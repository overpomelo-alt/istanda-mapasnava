## ⚠️ 布農族 App 適配說明（先讀這一段）

教師手帳是 Firebase Hosting + localhost 開發、布農族 App 是 GitHub Pages + Firebase Firestore + Google Apps Script。以下三條規則在布農族 App 環境下要做局部調整，**其餘 10 條完全照用**：

### 規則 12（標準工作流 ABCDE）— 布農族 App 版本

| 步驟 | 教師手帳 | 布農族 App |
|---|---|---|
| **A** | localhost 改 + 自測 | 在本機開檔案/或用 GitHub 線上編輯，自測 |
| **B** | localhost 親眼驗證 | **直接 push 到測試分支**，或在無痕視窗訪問 GitHub Pages 確認 |
| **C** | `firebase deploy --only hosting` | `git push origin main`（GitHub Pages 自動部署） |
| **D** | 線上 Ctrl+Shift+R 強制重整實測 | **同上**，GitHub Pages 通常 1-2 分鐘生效 |
| **E** | `git commit + push` | **已併入 C** |

布農族 App 通常只有 4 步：**A 改 → B 驗 → C 推上 GitHub → D 線上實測**。

### 規則 12 A 階段前置 — 布農族 App

不需要 firebase serve，但要確認：
- Firestore 規則目前是「test mode」、6/15 前必須改成正式規則（限制只讀/寫家人）
- Google Apps Script 端點要在無痕視窗能訪問（已驗證過一次，每次重新部署都要再測）

### 規則 13（加新資料變動來源、統計端要同步檢查）

對應到布農族 App 的「三條同步路徑」是：
- **Firestore 寫入端**（member.html 上傳錄音時、要寫進 `recordings` 集合）
- **Drive 寫入端**（同時寫進 Google Drive 資料夾）
- **首頁/個人頁面讀取端**（首頁 Feed、個人頁面清單、recordCount 統計都要能讀到）

加新功能（例如「按讚」「留言」「跟著學」）時，這三條路徑都要檢查。

## 規則本文（v1.3 原版 + 微調）

### 1. 數字要有名字、要集中
**❌ 錯誤示範：** 數字直接散在程式各處
```js
const progress = doneRecords / 87;        // 87 是什麼？
if (audioSeconds > 60) showWarning();
```

**✅ 正確做法：** 集中放在設定檔、給名字
```js
// config.js
export const FAMILY = {
  totalMembers: 87,
  maxAudioSeconds: 60,
};
// 使用時
const progress = doneRecords / FAMILY.totalMembers;
```

**理由：** 家族成員會增加、錄音長度限制可能調整。集中在一處方便更新一次，不用全檔搜尋。

**心法：** 看到數字問自己「**這個數字之後會不會變？**」會變的，給名字、放設定檔。

> ⚠️ **這條規則適用於新增功能。已經寫好且運作正常的舊程式碼，不要因為這條規則回頭重構。**

> 💡 **補充：不重構運作邏輯 ≠ 保留死代碼 / 死寫入。** 0 呼叫端的 function 或 0 讀取的物件屬性可以直接清。動工前嚴 grep（name / 動態存取 / 模板字串）、確認 0 讀取才動。

---

### 2. 寫通用函數、別重寫一次

**❌ 錯誤示範：** 每個地方都重複寫「base64 編碼上傳到 Apps Script」

**✅ 正確做法：** 寫 `uploadAudioToDrive(memberId, blob)`，各處複用。

**理由：** 重複的邏輯散在多處時、改一個地方很容易忘記改另一個。

---

### 3. 資料一處產生、多處使用

**理由：** 同樣的資訊（例如 recordCount）、不同地方獨立計算、會在某次修改時出現不一致。

**心法：** 想做新功能時、先看「現有的程式有沒有類似邏輯可以複用」、再決定要不要寫新的。

---

### 4. 防呆設計

**✅ 必要檢查：**
```js
if (!memberId) {                          // URL 沒帶 ID
  showError('沒有指定成員');
  return;
}
if (!memberData) {                        // Firestore 撈不到
  showError('找不到這位家人');
  return;
}
if (!recordings || recordings.length === 0) {   // 沒錄音時的 fallback
  showEmptyState();
}
```

**心法：** 寫每個函數時問自己：「**輸入會不會是空的？會不會超出預期？**」

---

### 5. 重置狀態 — UI 變化時、相關狀態都要清

**布農族 App 重點情境：**
- 錄音完成 → 試聽 → 重錄：要清掉 previewBlob、previewUrl、Audio 物件
- 上傳成功 → 隱藏試聽區、重新撈清單
- 換成員（從 Cina Umav 切到 Tama Iman）：要清掉所有播放中的音檔

**心法：** 設計 UI 互動時、想：「使用者離開又回來、看到的應該是什麼狀態？」

---

### 6. 不確定就問、不要硬寫

**✅ 好的做法：**
> 「Q1: 錄音上限要 60 秒還是 90 秒？」
> 「Q2: 重錄時舊錄音要保留在本機嗎？」

開發者攤開選項給使用者決定、再動手。

**❌ 不好的做法：** 看到規格不清楚、自己猜一個寫下去、結果跟使用者真實需求不符、白做工。

---

### 7. 規格設計時、考慮使用者真實工作流

**布農族 App 關鍵使用者畫像：**
- 1-90 歲全家族成員
- 重點長輩可能不熟 App 操作
- 年輕人/小朋友習慣 LINE、Instagram 互動

**心法：** 規格不是工程師的技術觀點、是「**家人實際使用時、會怎麼用？**」

---

### 8. 升級舊資料、不破壞舊功能

**✅ 正確做法：**
```js
// 例如未來加 photoUrl 欄位、舊資料沒有時自動補 null
members.forEach(m => {
  if (!m.photoUrl) m.photoUrl = null;
  if (!m.recordCount) m.recordCount = 0;
});
```

**心法：** 改資料結構時、想：「**家人打開 App 看到的不是空白、是他已經錄好的內容 — 怎麼讓舊資料平滑升級？**」

---

### 9. 改之前先讀完整脈絡

**❌ 錯誤示範：** 使用者說「改 X 函數」、Claude 直接打開 X 改了就交差。

**✅ 正確做法：** 改 X 函數之前、先：
1. 搜尋整個專案、看 X 在哪些地方被呼叫
2. 看 X 的回傳值被怎麼使用（顯示？計算？傳給別的函數？）
3. 確認改動後會不會影響到那些地方

**理由：** 函數很少獨立存在。改一個地方常常會牽動別處。

**心法：** 「**這個函數被誰用？用了什麼？我改了之後他們還能正常運作嗎？**」

---

### 10. 大功能先寫規格、再動工

**❌ 錯誤示範：** 邊做邊想、Claude Code 做一段、發現不對、又改、又確認、零碎來回十幾輪。

**✅ 正確做法：** 動工前、先跟規劃方（claude.ai 對話）討論透：
1. **把所有「等等再說」「之後再決定」全部攤開、定稿**、不留模糊
2. **寫 specs/XX.md**：把規格存到專案 specs 目錄
3. **規格裡放偵察報告格式**：Claude Code 第一步先偵察 + flag 風險、不直接動筆
4. **規格裡放 Step 順序**：Claude Code 不用自己想流程
5. **每個 Step 之間使用者回「ok」就好**：中途不插新要求

**理由：** 規格寫得越透、Claude Code 越能流暢推進、使用者省力。

**心法：** 「**這份規格、Claude Code 半年後沒這個對話也看得懂嗎？**」

> ⚠️ **這條規則適用於中大型功能。小修小補（改一行 CSS、加一個 console.log）不需要寫 specs。**

> 💡 **寫 spec 改動段時、要 grep 實際 code 確認、不能憑邏輯推測。**

---

### 11. 規則 1 跟規則 2 衝突時的判斷

當「動範圍」夠小、能不能抽出**只服務新需求的小工具**？

- **抽得出 → 走 A'（抽小工具、舊邏輯不動）** — 兩規則兼顧
- **抽不出來、要動到舊邏輯 → 規則 1 優先（複製一份、留 TODO 註解未來統一）**

**心法：** 「**舊程式不動的前提下、能不能用小工具讓新邏輯不重複？能就抽、不能就接受少量重複。**」

---

### 12. 標準工作流 ABCD — 不可跳順序（布農族 App 版）

| 步驟 | 內容 | 為什麼不能跳 |
|---|---|---|
| **A** | 本機/線上編輯器改 + 自測 | 沒測就推 push、上線壞了家人會踩 |
| **B** | 無痕視窗驗證（GitHub Pages 或 Apps Script） | Claude Code 自測通過 ≠ 真實 UI 流暢 |
| **C** | git push → GitHub Pages 自動部署 | — |
| **D** | 線上 Ctrl+Shift+R 強制重整實測 | 快取沒清、看到的是舊版、誤判 |

**任何一步失敗退回上一步、不前進。**

> 💡 **特別注意：Apps Script 改完程式碼，必須「管理部署作業 → 編輯 → 版本選新版本 → 部署」，否則部署的還是舊版。**

---

### 13. 加新資料變動來源時、三條路徑同步檢查

布農族 App 的「三條路徑」（見上方適配說明）：
- Firestore 寫入端
- Drive 寫入端
- 讀取端（首頁/個人頁面/recordCount 統計）

**心法：** 加新資料來源時、grep 看「現有讀這份資料的地方」、確認三條路徑都改到。

## Part 2 — Claude Code 權限與授權設定

### 推薦設定：Accept Edits 模式

#### 怎麼啟用
在 Claude Code 輸入框打：
```
/permissions
```
按 Enter → 選 **Accept Edits mode**

或用快捷鍵 **Shift + Tab** 切換模式。

#### 模式對比

| 模式 | 自動接受 | 才問你 | 適合 |
|---|---|---|---|
| **Default**（預設） | 無 | 每個編輯 | 第一次熟悉、需要每個都看 |
| **Accept Edits** ✨ | 專案內檔案編輯 + 讀取指令 | 危險指令、跑出專案 | 一般開發、推薦 |
| Plan mode | 不改檔案、純規劃 |（不需要）| 只想討論、不動工 |
| Bypass mode | 全部 | 從不問 | 不推薦、太危險 |

#### Accept Edits 自動接受的範圍

✅ **自動跑、不問你：**
- 讀檔指令：`grep / cat / ls / find / head / tail / wc`
- 專案內檔案編輯：index.html / member.html / config.js / specs/
- 重複出現的同一個提示

🟡 **還是會問你：**
- 修改類 bash 指令：`sed -i / rm / mv / >`
- 跑出專案範圍的操作
- 看起來危險的指令

---

### Part 2.5 — 提示三選一的判斷

#### 🟢 看到請選 1（Yes）

#### 🟡 不要選 2（永久允許）

**不要選 2** — 把判斷權交出去、後面萬一有意外就難挽回。

#### 🔴 看到請選 3（No）、停下來

| 情境 | 為什麼 |
|---|---|
| `rm -rf` 任何路徑 | 大規模刪除、極危險 |
| 動到 `~/.bashrc` 或系統檔 | 跑出專案範圍 |
| 連到外部網址下載執行（`curl ... | bash`） | 安全風險 |
| 看不懂的指令 | 不確定 = 暫停 |

選 3 後、貼給規劃方（claude.ai）看、討論再決定。
